Autograd: Backpropagation
from Scratch

⏱ ~60 min 🧠 Core Mechanism 💻 CPU only

How does PyTorch "know" how to update weights? This lesson tears open the black box. We will learn why gradient descent works, build backprop by hand, and see why Deep Learning is even mathematically possible. This is the engine that trains every neural network on Earth.

The Preamble: Why Weights and Calculus?

Before computing gradients, let's take a step back. If we have inputs (text) and targets (next word), what exactly are we learning? What do weights actually represent?

What Weights Actually Represent

Weights are statistical relationships. When a model reads "Paris is the capital of", the internal weights are literally structural numbers that say "If you see Paris, strongly boost the probability of France." W (Weights) are the volume sliders, and b (Biases) are the baseline offsets. Together, they are billions of learnable knobs.

So, we have billions of knobs. How do we tune them to get the right output? Historically, scientists tried several ways:

  • Random Guessing (Brute Force): Mathematically impossible. The universe would end before you found a good combination for 8 billion parameters.
  • Evolution/Probability: Make random mutations, see what survives. Better, but very slow because it's "blind" trial and error.
  • Calculus (Gradient Descent): Calculus gives us a compass. It allows us to ask a highly specific question: "If I turn this exact knob slightly to the right, does the error go up or down?"
🏔️
Why does gradient descent actually work?
Think of the loss function as a foggy mountain landscape. The "Gradient" is simply the direction of the steepest uphill incline from where you are standing. If you want to reach the bottom of the valley (minimum loss), you just step in the exact opposite direction of the gradient. You do this thousands of times. That is gradient descent.

The 5-Step Training Loop

If you break down the training of a neural network—whether it's a 2-parameter line or a 400-Billion parameter Llama 3 model—it always follows this exact 5-step loop:

The Universal Deep Learning Loop
# 1. FORWARD PASS: Pass data through the network
predictions = model(inputs)

# 2. COMPUTE LOSS: How wrong were the predictions?
loss = loss_function(predictions, targets)

# 3. BACKWARD PASS (Backprop): Calculate gradients for every weight
loss.backward()

# 4. UPDATE WEIGHTS: Step downhill (opposite of gradient)
optimizer.step()

# 5. REPEAT: Do this millions of times over the dataset

The Miracle of Backprop: Why Not Brute Force?

A very common beginner question: Why do we need complex calculus (Backprop)? Why can't we just tweak a weight by `0.001`, run the network, and see if the loss goes down?

That is called a Numerical Gradient. Here is why it is impossible for Deep Learning:

Numerical Gradient = ( Loss(W + 0.001) - Loss(W) ) / 0.001
To calculate this, you must run a full Forward Pass for EVERY SINGLE parameter.
✨
The $O(N)$ vs $O(1)$ Miracle
If you have a model like GPT-3 with 175 Billion parameters, calculating numerical gradients would require 175 Billion forward passes for a single training step.

Backpropagation (Analytical Gradients) computes the exact same thing using only 1 Forward pass and 1 Backward pass. Backprop is literally 175 billion times faster. Without it, Deep Learning simply would not exist.

What Exactly is Flowing Backward?

We say "gradients flow backward." But what does that physically mean? Think of every math operation in the network (a multiply, an add, a ReLU) as a Local Router Node.

During the Forward Pass, nodes receive Values, compute a result, and pass Values forward. During the Backward Pass, nodes receive Gradients from the right, multiply them by their local derivative (Chain Rule), and pass Gradients backward.

Animation: Values Forward (Green) / Gradients Backward (Gold)
The Chain Rule — in one sentence

If loss depends on y, and y depends on x, then how much does loss change when x changes? Answer: (∂loss/∂y) × (∂y/∂x). Backprop is just the Chain Rule applied iteratively through a graph.

Implement Backprop by Hand

Let's build the simplest possible neural network: one linear layer + MSE loss. We'll compute gradients manually first, then verify with PyTorch. This is the same pedagogical approach Andrej Karpathy uses in micrograd.

manual_backprop.py — no PyTorch autograd
MANUAL
import torch

# ── Our tiny model: y = w*x + b ───────────────────────────────
x      = 3.0   # input (fixed)
w      = 2.0   # learnable weight
b      = 1.0   # learnable bias
y_true = 5.0   # target

# ── FORWARD PASS (Values move right) ─────────────────────────
u      = w * x           # u = 6.0
y_pred = u + b           # y_pred = 7.0
loss   = (y_pred - y_true) ** 2  # loss = 4.0

# ── BACKWARD PASS (Gradients move left via Chain Rule) ───────

# Step 1: Base case (dL/dL = 1)
d_loss__d_loss = 1.0

# Step 2: loss = (y_pred - y_true)² -> derivative is 2*(y_pred - y_true)
d_loss__d_ypred = 2 * (y_pred - y_true)   # 2*(7-5) = 4.0

# Step 3: y_pred = u + b -> derivative of + is 1. Gradient routes perfectly to both.
d_loss__d_u = d_loss__d_ypred * 1.0       # = 4.0
d_loss__d_b = d_loss__d_ypred * 1.0       # = 4.0

# Step 4: u = w * x -> derivative w.r.t w is x.
d_loss__d_w = d_loss__d_u * x             # = 4.0 * 3.0 = 12.0

print(f"∂loss/∂w = {d_loss__d_w}")   # 12.0
print(f"∂loss/∂b = {d_loss__d_b}")   # 4.0

# ── VERIFY: PyTorch Autograd ──────────────────────────────────
w_t = torch.tensor([2.0], requires_grad=True)
b_t = torch.tensor([1.0], requires_grad=True)
x_t = torch.tensor([3.0])

loss_t = (w_t * x_t + b_t - 5.0) ** 2
loss_t.backward()

print(f"PyTorch ∂loss/∂w = {w_t.grad.item()}")  # 12.0 ✓
print(f"PyTorch ∂loss/∂b = {b_t.grad.item()}")  # 4.0  ✓

What does .backward() actually do internally?

PyTorch isn't magic. Under the hood, .backward() does exactly three things:

  1. It performs a Topological Sort on the computation graph (arranges all nodes from output back to input).
  2. It sets the gradient of the final node (loss) to 1.0.
  3. It loops through the sorted list in reverse, calling a local _backward() function on each node.
micro_engine.py — Demystifying the engine
INTERNALS
# If you wrote your own PyTorch, a Tensor object would look like this:
class Value:
    def __init__(self, data, _children=()):
        self.data = data
        self.grad = 0.0
        self._backward = lambda: None
        self._prev = set(_children)

    def __mul__(self, other):
        out = Value(self.data * other.data, (self, other))
        
        def _backward():
            # The Local Gradient Router! (Chain rule)
            self.grad += other.data * out.grad
            other.grad += self.data * out.grad
            
        out._backward = _backward
        return out
🧠
Why does PyTorch use so much GPU memory during training?
Look at the _backward() function above. To compute the gradient, the multiplier needs to "remember" the forward data (self.data and other.data). Therefore, Autograd must cache all intermediate forward tensors in memory so they are ready for the backward pass. This is why training takes 3x-4x more memory than inference!

A Real Mini-Network: The Matrix Dimension Trick

Going from scalars to vectors of millions of dimensions is where 90% of people get stuck. What is a gradient in a million dimensions? It's just a million-dimensional vector pointing downhill.

When working with Matrices, here is the secret "cheat code" used by neural network engineers: The shape of a gradient matrix must exactly match the shape of its original weight matrix.

Forward: h = ReLU(W₁·x + b₁), y = W₂·h + b₂
Deriving the backward pass step-by-step:

1. The Linear Layer (y = W₂·h): How do we find d_W2?
We know W₂ has shape (d_out, d_hidden). We receive d_y from above with shape (d_out, 1). We have the input h with shape (d_hidden, 1).
How do we combine (d_out, 1) and (d_hidden, 1) to get (d_out, d_hidden)?
Answer: The Outer Product! d_y @ h.T

2. The ReLU Valve (h = ReLU(z)): How do we find d_z?
ReLU is a literal physical valve. If the forward pass z was > 0, the valve is OPEN (derivative is 1, gradient passes). If z was ≤ 0, the valve is CLOSED (derivative is 0, gradient is permanently destroyed).
mlp_manual.py — matrix backprop
DEEP DIVE
import torch

# ── Network hyperparameters ───────────────────────────────────
d_in, d_hidden, d_out = 4, 8, 2

W1 = torch.randn(d_hidden, d_in)
W2 = torch.randn(d_out, d_hidden)
x  = torch.randn(d_in)         
d_z2 = torch.randn(d_out) # Fake upstream gradient from loss

# Forward
z1 = W1 @ x
h  = torch.relu(z1)

# Backward Matrix Math (Shape Matching!)
d_W2 = torch.outer(d_z2, h)       # (2,) ⊗ (8,) = (2, 8) matches W2!
d_h  = W2.T @ d_z2                # (8, 2) @ (2,) = (8,) matches h!

d_z1 = d_h * (z1 > 0).float()     # The ReLU Valve
d_W1 = torch.outer(d_z1, x)       # (8,) ⊗ (4,) = (8, 4) matches W1!

Batches, Transformers, and Vanishing Gradients

To finish bridging the gap to modern LLMs, we need to answer three massive questions.

1. Why do we average gradients over Mini-Batches?

Instead of calculating the gradient for 1 sentence, we calculate it for 32 sentences simultaneously, sum the gradients, and divide by 32. Why? First, GPU matrix multiplication is vastly more efficient in blocks. Second, a single sentence is "noisy" and might point the gradient in a weird direction. Averaging stabilizes the compass.

2. Why do deep networks cause Vanishing/Exploding Gradients?

Because the Chain Rule is just multiplication! If you have a 100-layer network, backprop will multiply 100 fractions together. 0.9 × 0.9 × 0.9 ... = 0.00002 (Vanishing gradient; early layers never learn). If it's 1.1 × 1.1 ... = 13,780 (Exploding gradient; network collapses to NaN). This is exactly why Transformers use Residual Connections—they act as gradient highways that bypass the multiplication!

3. How does this scale to Transformers?

Backprop in Llama 3 operates identically to our tiny script above, just scaled up. The flow of gradients goes: Loss → Logits → Final LayerNorm → Attention Block 32 → MLP Block 32 → ... → Attention Block 1 → Embedding Matrix. Every single weight matrix routes the gradient backward via outer products.

⚠️
The Gradient Accumulation Trap: .backward() ACCUMULATES gradients by default. If you don't call optimizer.zero_grad() at the start of step 1, step 2 will literally add its gradients on top of step 1's gradients. Your model will train on garbage.

torch.no_grad() — Disabling the Engine

When you are generating text (Inference), you aren't training. You don't need gradients. Use torch.no_grad() to tell PyTorch to stop tracking the forward graph. This frees up massive amounts of VRAM.

no_grad_usage.py
IMPORTANT
with torch.no_grad():
    y_infer = model(x)
    print(y_infer.grad_fn)  # None — no graph, less memory!
