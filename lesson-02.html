How does PyTorch "know" how to update weights? This lesson tears open the black box completely. We derive why gradients exist, implement backprop step-by-step with scalar math, build a tiny autograd engine from scratch, and understand the computational miracle that makes modern AI physically possible.

01 — Why Do We Need Gradients?

Before computing a single gradient, answer the prior question: why do gradients exist at all? We have a model with billions of weights. They need to change in the right direction to reduce loss. The gradient tells each weight exactly which direction that is — and by how much.

❌ Random Guessing
Randomly change weights and re-check loss. Mathematically impossible at scale. With 8 billion parameters, the universe ends before you find a good combination by chance.
⚠ Numerical Gradients
Nudge each weight by +0.001, re-run the network, measure the loss change. Mathematically correct — but requires one full forward pass per parameter. Covered in section 04.
✓ Backpropagation
Use the chain rule to compute the gradient for every parameter simultaneously in one backward pass. Same result as numerical gradients. 175 billion times faster. All of modern AI runs on this.
🏔️
The mountain metaphor. The loss function is a high-dimensional foggy landscape. You're standing somewhere on it, wanting to reach the valley (minimum loss). The gradient is the direction of steepest ascent from your current position. Step in the opposite direction — that's gradient descent. Repeat thousands of times across the entire dataset. That is the entirety of neural network training.

Concretely: a large positive gradient on weight w means "if w increases, the loss increases a lot — so decrease w." Near-zero gradient means "this weight barely affects the loss right now — leave it alone." The optimizer reads these signals and moves every weight toward lower loss simultaneously.

02 — The Universal Training Loop

Every neural network ever trained — from a 2-parameter linear model to Llama-3 with 70 billion parameters — runs this exact same 5-step cycle, millions of times. Understanding each step is understanding all of training.

The 5-Step Cycle — Repeated Millions of Times
1
Forward Pass — predictions = model(x)
Data flows through the network layer by layer. Each layer transforms its input: h = relu(W·x + b). Values cascade from input → hidden → output. PyTorch silently records every single operation in a computation graph, ready for step 3.
2
Compute Loss — loss = loss_fn(predictions, targets)
Compare predictions to ground truth. The output is a single scalar — the loss. Lower is better. For language models we use cross-entropy. This single number is what all of backpropagation flows backward from.
3
Backward Pass — loss.backward()
PyTorch walks the recorded computation graph in reverse, applying the chain rule at each node. Every learnable parameter accumulates its gradient in its .grad attribute. This is the step that makes deep learning possible — all gradients in one pass.
4
Update Weights — optimizer.step()
The optimizer reads each param.grad and adjusts the weight: w ← w − lr × ∂L/∂w. Step in the direction that reduces loss. This is the only moment weights actually change during training.
5
Zero Gradients — optimizer.zero_grad()
PyTorch accumulates gradients by default — it adds to existing .grad, never replaces it. You must clear gradients before the next step or they compound incorrectly. The most common beginner bug.
⚠️
The most common training bug. Forgetting optimizer.zero_grad(). Without it, gradients from step 1 are still in param.grad. Step 2 adds on top. The effective gradient grows each step until weights explode and loss → ∞. Always call it at the start of each training step, before the forward pass.

03 — What a Gradient Actually Is

Most beginners think "gradient = a single number." That is only true for a function of one variable. In neural networks with millions of parameters, a gradient is a vector with one number per parameter — each telling you how much the loss changes if that specific parameter increases by a tiny amount.

Simple: 1 parameter
f(w) = w²one parameter
∂f/∂w = 2wone gradient value
scalara single number
Neural Net: N parameters
L(w₁,w₂,…)N parameters
∇L = [∂L/∂w₁, ∂L/∂w₂, …]N values
vectorsame shape as weights
The Shape Rule — Always

The gradient of the loss with respect to any weight tensor has exactly the same shape as that weight tensor. A weight matrix of shape (4096, 4096) produces a gradient of shape (4096, 4096). param.grad is always the same shape as param. This is how PyTorch knows where every gradient goes — no ambiguity, no allocation needed beyond what already exists.

For Llama-3-70B with 70 billion parameters, the full gradient is a 70-billion-dimensional vector — one number per weight — computed simultaneously in a single backward pass. That this is possible at all is the point of the next section.

04 — The Miracle: Why Backprop Exists

A natural question: "Why can't we just nudge each weight by +0.001, re-run the network, and check if loss goes down?" This is a numerical gradient. Mathematically perfect. Computationally impossible.

numerical_gradient.py — correct, but catastrophically slow
INTUITION
def numerical_grad(model, loss_fn, x, y, param, eps=1e-4):
    param.data += eps
    loss_up    = loss_fn(model(x), y)   # ← full forward pass #1
    param.data -= 2 * eps
    loss_down  = loss_fn(model(x), y)   # ← full forward pass #2
    param.data += eps
    return (loss_up - loss_down) / (2 * eps)

# Cost: 2 full forward passes PER parameter.
# GPT-3: 175B parameters → 350 billion forward passes for ONE training step.
# Backpropagation: 1 forward + 1 backward = SAME gradients, all params.
Numerical Gradients (GPT-3)
350B
forward passes per training step. At 1 ms per pass: 11 years per gradient update. Deep learning would be physically impossible.
Backpropagation (GPT-3)
2
passes per training step. One forward, one backward. Identical gradients. This is the computational miracle that made the last decade of AI possible.
✨
Why is backprop so fast? The chain rule lets you reuse intermediate derivatives. Instead of recomputing gradients from scratch for each parameter, backprop computes each node's local derivative once during the backward pass and multiplies it with the incoming gradient. Total cost is proportional to the number of operations in the graph — same as the forward pass — regardless of how many parameters there are.

05 — What Physically Flows Backward

Think of every operation — a multiply, an add, a ReLU — as a local routing node. Forward pass: receives values, computes result, passes values right. Backward pass: receives a gradient from the right, multiplies it by the node's local derivative (chain rule), routes the gradient left to its inputs.

The Chain Rule — One Sentence

If loss depends on y, and y depends on x: ∂loss/∂x = (∂loss/∂y) × (∂y/∂x). Each node multiplies the incoming gradient by its own local derivative and passes the result backward. Backpropagation is exactly this rule, applied node by node, from loss all the way back to inputs.

06 — Implement Backprop by Hand

The fastest path to understanding autograd: compute gradients manually, then verify with PyTorch. We use a tiny one-layer network — y = w*x + b, loss = MSE — the same example Karpathy uses in micrograd because no complexity hides the math.

manual_backprop.py — hand computation then PyTorch verification
CORE CODE
import torch

# ── Network: y = w*x + b,  Loss = (y - target)² ──────────────
x      = 3.0   # input (fixed, not learnable)
w      = 2.0   # learnable weight
b      = 1.0   # learnable bias
target = 5.0   # ground truth

# ── FORWARD: values flow right ────────────────────────────────
u      = w * x                       # 2.0 × 3.0 = 6.0
y_pred = u + b                       # 6.0 + 1.0 = 7.0
loss   = (y_pred - target) ** 2     # (7.0 - 5.0)² = 4.0

# ── BACKWARD: chain rule, node by node ───────────────────────

# Base case: dL/dL = 1 (always, trivially)
dL_dL     = 1.0

# loss = (y_pred - target)²   →   d/dy_pred = 2*(y_pred - target)
dL_dypred = 2 * (y_pred - target)    # 2*(7-5) = 4.0

# y_pred = u + b   →   both inputs get the full incoming gradient (local deriv = 1)
dL_du     = dL_dypred * 1.0           # 4.0
dL_db     = dL_dypred * 1.0           # 4.0

# u = w * x   →   du/dw = x
dL_dw     = dL_du * x                 # 4.0 × 3.0 = 12.0

print(f"Manual  ∂L/∂w = {dL_dw}")   # 12.0
print(f"Manual  ∂L/∂b = {dL_db}")   # 4.0

# ── VERIFY with PyTorch autograd ─────────────────────────────
w_t = torch.tensor([2.0], requires_grad=True)
b_t = torch.tensor([1.0], requires_grad=True)
x_t = torch.tensor([3.0])  # no grad on inputs

loss_t = (w_t * x_t + b_t - 5.0) ** 2
loss_t.backward()

print(f"PyTorch ∂L/∂w = {w_t.grad.item()}")  # 12.0 ✓
print(f"PyTorch ∂L/∂b = {b_t.grad.item()}")  # 4.0  ✓
✓
Run this yourself. Copy it, execute it, watch the numbers match. PyTorch is doing the exact same arithmetic you just did by hand — just automatically, for every parameter simultaneously, in optimized C++/CUDA.

07 — Build a Tiny Autograd Engine

The cleanest way to permanently demystify autograd: build a mini version of it yourself. This is Karpathy's micrograd idea. Each Value wraps a number, stores a gradient, and carries a _backward function that knows how to route gradients through its specific operation. That's literally all PyTorch tensors are.

micro_autograd.py — the most important 40 lines in this course
class Value:
    def __init__(self, data, _parents=()):
        self.data      = data
        self.grad      = 0.0            # accumulates during backward
        self._backward = lambda: None   # set by each operation
        self._prev     = set(_parents)  # inputs that created this node

    def __mul__(self, other):
        out = Value(self.data * other.data, (self, other))
        def _backward():
            # Chain rule: incoming_grad × local_derivative
            self.grad  += other.data * out.grad  # dL/d_self  = other × dL/d_out
            other.grad += self.data  * out.grad  # dL/d_other = self  × dL/d_out
        out._backward = _backward
        return out

    def __add__(self, other):
        out = Value(self.data + other.data, (self, other))
        def _backward():
            self.grad  += 1.0 * out.grad   # addition routes full gradient to both inputs
            other.grad += 1.0 * out.grad
        out._backward = _backward
        return out

    def backward(self):
        # 1. Topological sort: leaves first, loss node last
        topo, visited = [], set()
        def build(v):
            if v not in visited:
                visited.add(v)
                for p in v._prev: build(p)
                topo.append(v)
        build(self)
        # 2. Walk backward, calling _backward() at each node
        self.grad = 1.0                    # seed: dL/dL = 1
        for node in reversed(topo):
            node._backward()

# ── Verify matches our manual calculation ─────────────────────
x = Value(3.0);  w = Value(2.0);  b = Value(1.0)
diff = (w * x + b) + Value(-5.0)
loss = diff * diff
loss.backward()
print(f"w.grad = {w.grad}")  # 12.0 ✓  same as manual, same as PyTorch
print(f"b.grad = {b.grad}")  # 4.0  ✓
💡
This is all of PyTorch autograd. A much faster, GPU-accelerated version of exactly this class. Every torch.Tensor with requires_grad=True has a grad_fn — its _backward function. When you call loss.backward(), PyTorch performs the same topological sort, sets loss.grad = 1.0, and walks the graph calling each node's grad_fn.

08 — Why Training Uses 3–4× More Memory Than Inference

Look at the _backward function for multiplication: it needs other.data — the value from the forward pass — to compute the gradient. This means PyTorch must keep every intermediate activation from the forward pass alive in memory until the backward pass is finished using it.

ModeWhat Stays in MemoryWhyApprox Cost
InferenceWeights + one layer at a timeNo backward needed, activations freed immediately~1× weights
TrainingWeights + ALL forward activations_backward() needs them to compute gradients~3–4× weights
Training + Adam+ momentum m, variance v per paramAdam stores running statistics per parameter~6× weights
no_grad.py — disable the graph during inference
IMPORTANT
import torch

# Inference: don't build the computation graph
model.eval()
with torch.no_grad():
    output = model(x)
    print(output.grad_fn)   # None — graph not built, 3-4× less VRAM

# gradient_checkpointing: the training memory tradeoff.
# Instead of saving ALL activations from forward, save only "checkpoints"
# at layer boundaries and RECOMPUTE the rest during backward.
# Cost: ~33% extra compute. Benefit: ~4× less activation memory.
model.gradient_checkpointing_enable()   # used in all LLM fine-tuning

09 — Scaling to Matrices: The Shape Trick

Everything above uses scalars. Neural networks use matrices. The transition has one key rule: the gradient always has the same shape as the original tensor. Once you know the target shape, you can often derive the correct operation purely from shape algebra — without working through the full calculus.

mlp_manual_backprop.py — matrix gradients via shape matching
DEEP DIVE
import torch

d_in, d_hidden, d_out = 4, 8, 2
W1 = torch.randn(d_hidden, d_in)   # shape (8, 4)
W2 = torch.randn(d_out, d_hidden)  # shape (2, 8)
x  = torch.randn(d_in)             # shape (4,)

# ── Forward ───────────────────────────────────────────────────
z1 = W1 @ x          # (8,4)@(4,) = (8,)   pre-activation
h  = z1.relu()        # (8,)                ReLU gate
z2 = W2 @ h          # (2,8)@(8,) = (2,)   output

dL_dz2 = torch.randn(d_out)  # upstream gradient from loss

# ── Backward: shape matching reveals the operations ──────────

# Need dL/dW2 with shape (2,8) to match W2.
# We have dL_dz2 (2,) and h (8,).
# (2,) outer (8,) = (2,8) ✓
dL_dW2 = torch.outer(dL_dz2, h)       # (2,8) — matches W2
dL_dh  = W2.T @ dL_dz2               # (8,2)@(2,) = (8,) — matches h

# ReLU valve: gradient is 0 where z1 ≤ 0, unchanged where z1 > 0
dL_dz1 = dL_dh * (z1 > 0).float()    # zeros dead neurons permanently

# Need dL/dW1 with shape (8,4) to match W1.
# dL_dz1 (8,) outer x (4,) = (8,4) ✓
dL_dW1 = torch.outer(dL_dz1, x)       # (8,4) — matches W1
ReLU: The Gradient Valve

ReLU is max(0, x). Its derivative: 1 when x > 0, 0 when x ≤ 0. During backprop it acts as a binary valve — gradient flows through unchanged if the forward value was positive, and is zeroed out forever if the forward value was negative. A neuron that always outputs negative values receives zero gradient every step. This is the "dead neuron" problem.

10 — Deep Networks, Vanishing Gradients & Transformers

The chain rule is multiplication. A 100-layer network multiplies 100 local gradients together during backprop. If each is slightly less than 1, the product approaches zero exponentially — early layers receive almost no gradient and never learn. If each is slightly greater than 1, gradients explode.

❌ Vanishing
0.9¹⁰⁰ = 0.000026

Early layers get ~zero gradient. They stop learning entirely. Only the last few layers train effectively.
❌ Exploding
1.1¹⁰⁰ = 13,780

Gradients grow exponentially. Weights update by enormous amounts. Loss diverges to NaN. Clip with clip_grad_norm_.
✓ Residual Fix
x = x + layer(x)

Gradient flows through the identity path (+) without multiplication. This is why transformers use residual connections. Not just a convenience — a mathematical necessity.
Where Backprop Flows in nanoLlama

When loss.backward() runs on nanoLlama, the gradient flows: cross-entropy → logits → output head (weight-tied with embedding) → final RMSNorm → transformer block 4 → transformer block 3 → transformer block 2 → transformer block 1 → token embedding. Residual connections in every block ensure the gradient reaches the embedding without vanishing. Every weight matrix in every block receives its gradient from this single pass.

11 — The Real Test: 7 Questions

These are the questions that reveal whether you understand backprop or just its mechanics. Hover any card to reveal the key insight.

  • Q1
    Why is backpropagation computationally efficient for neural networks with billions of parameters?
    The chain rule lets you reuse intermediate derivatives. Each node computes its local derivative once and multiplies it with the incoming gradient. All N parameter gradients are produced in one backward pass — not N forward passes. Cost is proportional to the number of operations, not the number of parameters.
    ← hover to reveal
  • Q2
    A neuron computes z = −5.0, passes through ReLU. Upstream gradient is 10.0. What gradient flows backward through the ReLU?
    Zero. ReLU's derivative is 0 when z ≤ 0. The gradient is permanently blocked — this neuron is "dead." It will receive zero gradient from every backward pass until some upstream change makes z positive again.
    ← hover to reveal
  • Q3
    Weight matrix W has shape (64, 128). What is the shape of dL/dW?
    (64, 128) — exactly the same as W. The gradient always has the same shape as the tensor it's the gradient of. This is non-negotiable. It's how PyTorch allocates param.grad — same shape, no ambiguity.
    ← hover to reveal
  • Q4
    You call loss.backward() and optimizer.step() each step. After 10 steps, loss explodes to infinity. What single line of code is missing?
    optimizer.zero_grad(). Without it, gradients accumulate across steps. By step 10 the effective gradient is 10× what it should be. Weights take enormous steps, loss diverges. Must call zero_grad() before each forward pass.
    ← hover to reveal
  • Q5
    Why does training a model use 3–4× more GPU memory than running inference with the same model?
    Because .backward() needs the intermediate activations from the forward pass to compute gradients (they appear in each node's _backward function). All activations are kept alive from forward pass until their corresponding backward computation is done. Inference frees each activation immediately after use.
    ← hover to reveal
  • Q6
    A 100-layer network has vanishing gradients. What architectural change fixes this without changing learning rate or activation?
    Residual connections: x = x + layer(x). The gradient flows directly through the identity branch (addition distributes gradient with scale 1) without any multiplication that could shrink it. This is the entire purpose of residuals in transformers and ResNets.
    ← hover to reveal
  • Q7
    Node z = a + b. Upstream gradient is 3.0. What gradient does each input (a and b) receive?
    Both receive 3.0. Addition's local derivative is 1 for both inputs, so the incoming gradient passes through unchanged and in full to both branches. This is why gradient "accumulates" at addition nodes — both inputs get the complete upstream signal.
